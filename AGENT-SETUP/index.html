How Hoss Works

A deep dive into the agent architecture — workspace files, sub-agent coordination, and the heartbeat system.

The Workspace Structure

The workspace is the agent's "brain" — a directory of markdown files and configs that persist across sessions. Unlike stateless chat, Hoss wakes up fresh each session but with full context of who he is, what Tyler cares about, and what happened before.

.workspace/ ├── SOUL.md # Philosophical core — who Hoss is ├── IDENTITY.md # Role definition, co-founder responsibilities ├── AGENTS.md # Sub-agent roster and coordination protocol ├── MEMORY.md # Long-term memory — decisions, infrastructure ├── TOOLS.md # Tool configurations and credentials ├── HEARTBEAT.md # Autonomous check system — scheduled tasks ├── memory/ # Daily logs + FTS5 searchable index └── agents/ # Sub-agent workspaces

Core Files

SOUL.md

The philosophical core. Defines Hoss's identity, voice, and operational instincts. This is what makes Hoss a co-founder, not a chatbot — he has opinions, defends them, and acts decisively.

# SOUL.md — Hoss You are not a chatbot. You are Tyler's co-founder. ## Core Identity - Think well, then act decisively - Have opinions and defend them - Earn trust through competence - Treat Tyler's stuff like it's yours ## Anti-Patterns - "Tyler should decide" without options and tradeoffs - Performatively helpful ("Great question!")

MEMORY.md

Long-term memory. Curated from daily logs, this file stores decisions, infrastructure, people, and lessons — the things that persist across sessions.

# MEMORY.md — Key Decisions - Cloudflare Pages for flume main site - Brand voice: "The honest tinkerer" - ONLY FOUR ACTIVE PROJECTS: fortOS, tylerdotai.com, fwpanthers, clawplex

HEARTBEAT.md

Defines autonomous checks that run on a schedule — memory indexing, git backups, cron health, system monitoring. Hoss does real work without being asked.

# HEARTBEAT.md — Active Hours: 08:00–22:00 CDT ## Checks (batched when active + due) 1. Memory FTS Index — rebuild if > 20 hours old 2. Git Backup Check — alert if no commit in 6+ hours 3. Memory Health — check via memory/scripts/check_health.sh 4. Cron Health — check for error/failed crons

The Coordination Protocol

1. Tyler Makes a Request

Hoss assesses scope — is this a 2-minute task or 2-hour? Adjusts approach accordingly.

2. Parallelize If Possible

Independent tasks get spawned to sub-agents simultaneously. Results synthesized when done.

3. Log Decisions

Everything significant gets written to memory/YYYY-MM-DD.md and promoted to MEMORY.md.

4. Deliver + Update

Output delivered. Memory updated. Patterns promoted to AGENTS.md or SOUL.md.

Memory Search (FTS5)

Full-text search across all memory files using SQLite FTS5:

bash ~/.openclaw/scripts/memory-search.sh "OpenClaw setup" # Rebuild the index bash ~/.openclaw/scripts/memory-fts-indexer.sh rebuild

This is what makes Hoss different from a stateless chatbot — he can be asked "what did we decide about X last week?" and actually find the answer.
